# ENS (EventNodeSystem) 重构 Phase 3: Tri-Inspector 接入与编辑器数据链路打通

## 0. 前言（与 Phase2 的衔接）
Phase2 已完成运行时底座：EventSequence、ENSRegistry、指令指针执行、Label/Jump、死循环防护与任务级并发调度。Phase3 的职责是把编辑器链路彻底打通，使创作者在 Inspector 中可以直观、可靠地编辑多态化序列与语法糖（尤其是 If 语法糖），为后续 UI 编辑器（Phase4）与自动预编译（If -> Label/Jump 展开）奠定数据基础。

本文件为 Phase3 的正式规划与当前进展记录：明确能力边界、具体可交付成果、验收标准与实施步骤，并列出必须与 Phase2 保持兼容的要点。

---

## 1. 目标与边界（保持不变）
- 使用 Tri-Inspector 为 `[SerializeReference]` 的 `EventSequence.commands` 提供稳定的多态创建与字段展开能力。
- 为 If 语法糖提供编辑器级别的桥接（IfEditorBridge），支持在 Inspector 中编辑 If 的条件组、分支块与结束标记，能正确序列化为 `IfData`，并带有足够信息供预编译器转换为 Label/Jump 结构。
- 为常用 Condition/Action Data 补齐可编辑字段与默认值，提升编辑体验与一致性。

边界（Phase3 不做）：
- 不修改或替换 Phase2 的运行时执行引擎（Jump/Label 为唯一底层跳转）。
- 不实现 UI Toolkit 的独立事件页窗口（留给 Phase4，但 Phase3 应保证数据与小型 API 可被复用）。
- If 仍为编辑器语法糖，运行时仅识别 Label/Jump（If 的展开由预编译器或运行时扩展完成）。

   - 采用外部平铺条件组的方式组织条件表达式，通过 `jumpId / conditionGroupId / blockId / anchorId` 等标识与 Jump 绑定。

   - 仅保留模板元数据、块边界与展开规则，不再以嵌套编辑为目标。

5. 编辑器导出与验证
   - 提供“验证”按钮/命令：在 Editor 下对当前 `EventSequence` 进行静态检查，至少校验：
	 - Jump 指向的 Label 是否存在（警告/错误并包含序列索引与标签名）。
	 - Label 重名检测（错误）。
	 - 条件组与 Jump 的绑定是否完整。
	 - 检测可能导致死循环的直观跳转（静态启发式检查，不能替代运行时检测）。
   - 保存时保证模板展开结果能被 Phase4 或预编译脚本识别并展开为 `Label + BranchJump + EndJump` 的结构。

6. 可拓展性

## 4.1 Jump 与条件的代码级绑定规则（定案）

### 设计结论
- **不引入复合条件节点树。**
- **每个 Jump 只捕捉其后紧邻的一个条件节点。**
- 条件节点只负责单一判断；更复杂的逻辑通过多个 Jump 串联实现。
- **End of Condition 不采用。**

### 代码层约定
1. **Jump 的 lookahead 规则**
   - 在解析或执行 Jump 时，只检查 `Commands[index + 1]`。
   - 若该节点是 `Condition` 基类子类，则将其视为当前 Jump 的条件数据。
   - 若下一节点为空、越界或不是 `Condition`，则当前 Jump 按“无条件跳转”处理。

2. **条件节点的消费规则**
   - 被 Jump 捕捉到的条件节点仅作为附着数据，不应再按独立指令执行。
   - 执行器在完成 Jump 判定后，必须跳过该条件节点，避免重复执行或重复消耗。
   - 条件节点不需要额外的结束节点来界定范围。

3. **缺失/异常情况处理**
   - `Commands[index + 1]` 为空：按无条件 Jump 处理，且在编辑器校验阶段给出提示（若该 Jump 被配置为“必须绑定条件”则报错）。
   - `Commands[index + 1]` 不是 Condition：按无条件 Jump 处理，并在校验阶段警告“后继节点未被识别为条件节点”。
   - 条件节点存在但其数据不完整：执行阶段应视为条件失败或直接回退为不跳转，并输出可定位日志。

### 方案 A：Jump 后跟随条件块并依赖 End of Condition
**状态**：不采用，保留为历史设计参考。

**不采用原因**
- 需要 End 标记来界定范围，编辑器和保存流程都要额外维护边界。
- 一旦漏放 End 节点，序列结构会变得歧义，解析复杂度上升。
- 该方案比单条件绑定更重，不符合当前“简洁、平铺、单一职责”的目标。

### 方案 B（定案）：Jump 后仅跟随一个条件节点，不再使用 End of Condition
**思路**：Jump 本体后只允许紧跟一个条件节点，该条件节点就是唯一判断依据，不再允许多个条件节点组成块，也不需要 End 标记。

**优点**
- 结构最简洁，Jump 的语义最清晰。
- 编辑器无需维护块范围，保存与校验规则更简单。
- 每个条件节点只承担单一职责，更容易做默认值、比较符号和显示摘要。
- 与“多个 Jump + 单条件”表达复杂逻辑的思路一致。

**缺点**
- 单个条件节点的表达能力要靠额外配置项补足。
- 如果未来需要块级条件逻辑，可能需要重新引入边界标记。

### 条件能力的实现取舍
- **弃用复合条件**，不再追求 All/Any、条件树、条件组等复合结构。
- 改为扩展每个条件节点的配置项，让单节点具备更多比较能力。
- 例如 `PlayerHasAttributeData` 可扩展为支持：
  - 大于
  - 小于
  - 大于等于
  - 小于等于
  - 等于
  - 不等于
- 这样既能覆盖单条件取反/变体，也能用多个 Jump 串联表达更复杂逻辑，而无需维护复合条件系统。

### 方案建议
- **默认采用方案 B。**
- 如果后续发现某些场景确实需要块级条件，再单独评估是否重新引入边界标记，但不作为本阶段默认设计。

## 4.2 If/For 的建模方案与去嵌套路径（定案）

### 弃用方案 A：If/For 作为真正的节点，内部持有子节点列表
**状态**：弃用，不再作为后续实现方向。

**弃用原因**
- 会引入嵌套，和当前“Jump 是唯一底层实现”的目标不一致。
- 编辑器复杂度高，Tri 对多层 SerializeReference 展开支持有限。
- 运行时与预编译之间的边界容易变模糊。

### 方案 B（定案）：If/For 作为“预制 Jump 模板”或“编译宏”
**思路**：If/For 不保留为运行时控制树节点，而是在编辑器阶段被展开成平铺的 `Label + 条件 Jump + End Jump` 结构；If/For 只负责承载元数据和生成规则。

**优点**
- 完全避免嵌套，和当前线性事件表的设计一致。
- 运行时仍然只有 Jump/Label，底层模型最稳定。
- 编辑器可以逐步演进：先做模板，再做可视化高亮与验证。
- 便于统一条件系统，也方便未来增加其它语法糖。

**缺点**
- 需要明确模板展开规则和块边界表示方式。
- 用户可能会觉得 If/For 不是“真正的节点”，学习成本略高。
- 调试时需要额外的预编译映射信息来回看模板与展开结果。

### 是否能避免嵌套？
- **可以**。如果要保留 If/For 这个概念，建议不要让它们持有真正的子节点列表，而是让它们成为：
  1. 编辑器可识别的模板头；
  2. 由外部平铺命令构成的块范围；
  3. 预编译时转换成 Jump/Label 的宏。

### 推荐结论
- **默认把 If/For 定义为编辑器侧的预制模板，而不是运行时容器节点。**
- 如果后续确实要做成“节点”，也建议只保留“锚点 + 块范围 + 元数据”，不要持有嵌套子列表，以便最终仍然落回平铺序列。

## 7. 风险与缓解（补充）
- 风险：Tri 插件在不同 Unity 版本或项目配置下差异导致展示异常。  
  缓解：封装 Tri 的调用点（TriInspectorBridge），并提供无 Tri 的回退路径（最小化功能：只显示序列摘要与只读字段）。
- 风险：单条件节点配置项过多，导致节点语义膨胀。  
  缓解：保持每个条件节点的配置项只覆盖同一类比较语义，不把多个判断域塞进同一节点。
- 风险：历史数据或模板缺失结束边界标记。  
  缓解：新数据不生成边界标记；历史数据在保存时自动折叠为兼容结构，校验时若发现结构不完整则强制报错并给出具体索引与建议修复方式。
- 风险：Jump 节点表达式语法与运行时不匹配。  
  缓解：在实现前先与运行时接口（EventRunnerService）定义明确的表达式序列化协议，并通过单元测试与示例 Sequence 验证两端一致性。
