# ENS (EventNodeSystem) 重构 Phase 3: Tri-Inspector 接入与编辑器数据链路打通

## 0. 前言（与 Phase2 的衔接）
Phase2 已完成运行时底座：EventSequence、ENSRegistry、指令指针执行、Label/Jump、死循环防护与任务级并发调度。Phase3 的职责是把编辑器链路彻底打通，使创作者在 Inspector 中可以直观、可靠地编辑多态化序列与语法糖（尤其是 If 语法糖），为后续 UI 编辑器（Phase4）与自动预编译（If -> Label/Jump 展开）奠定数据基础。

本文件为 Phase3 的正式规划与当前进展记录：明确能力边界、具体可交付成果、验收标准与实施步骤，并列出必须与 Phase2 保持兼容的要点。

---

## 1. 目标与边界（保持不变）
- 使用 Tri-Inspector 为 `[SerializeReference]` 的 `EventSequence.commands` 提供稳定的多态创建与字段展开能力。
- 为 If 语法糖提供编辑器级别的桥接（IfEditorBridge），支持在 Inspector 中编辑 If 的条件组、分支块与结束标记，能正确序列化为 `IfData`，并带有足够信息供预编译器转换为 Label/Jump 结构。
- 为常用 Condition/Action Data 补齐可编辑字段与默认值，提升编辑体验与一致性。

边界（Phase3 不做）：
- 不修改或替换 Phase2 的运行时执行引擎（Jump/Label 为唯一底层跳转）。
- 不实现 UI Toolkit 的独立事件页窗口（留给 Phase4，但 Phase3 应保证数据与小型 API 可被复用）。
- If 仍为编辑器语法糖，运行时仅识别 Label/Jump（If 的展开由预编译器或运行时扩展完成）。

## 2. 当前进展（新增状态记录）
1. Tri-Inspector 状态（已完成）
   - Tri-Inspector 已导入并在项目中启用。
   - 在 Inspector 中，`EventSequence` 的 `commands` 列表已能通过 Tri 的多态下拉菜单直接创建不同的 `BaseNodeData` 子类实例，并能在展开信息中修改子项的数据（字段可编辑、Tooltip 与默认值展示正常）。
2. If/Condition 编辑器（已知限制）
   - 目前仍存在无法在 Inspector 中直接编辑内嵌在 `IfData`（或 If 节点）中的 `ConditionNode` 的问题：Tri 对内嵌复杂对象的展开存在限制，导致 If 内部的 Condition 条目只能以只读或摘要形式展示，不能像顶层 `commands` 那样直接通过 Tri 展开字段并编辑。
   - 需要通过 IfEditorBridge 提供自定义渲染/表单，绕过 Tri 仅展开第一层的限制，支持对 Condition 的完整编辑（条件组合、取反、成员字段等）。
   - 当前进展补充：Tri 在顶层列表工作良好，但 If 内嵌 Condition 的编辑依赖 IfEditorBridge 的实现。该项为 Phase3 的首要阻塞点。
3. 与 Phase2 的衔接需求（进行中）
   - 计划将 Jump 节点扩展为支持复杂条件表达式（见 Phase2 指南关于 Flow/Jump 的设计注记）。
   - 该扩展要求编辑器端能够创建/编辑复杂条件表达式并将其序列化到 Jump/Label/IfData 中，编辑器与运行时之间需要就表达式语法、序列化格式和预编译约定达成一致。
   - 已将 Phase2 的相关未决事项迁移到本文件的 TODO 清单，便于在 Phase3 实施时逐项处理或转交负责运行时的同学完成协同工作。

## 4. Phase3 编辑器能力清单（更新版）
1. Tri-Inspector 集成
   - Tri 已导入并可用；在 `EventSequence.commands` 上使用 Tri 的多态列表特性，保证用户能在 Inspector 中创建并展开顶层子类。
   - 为 `BaseNodeData` 子类提供 `GetSummary()` 支持，Tri 列表能显示摘要行（已完成/正在补充摘要内容）。

2. IfEditorBridge（关键能力，优先级最高）
   - 为 `IfData` 提供自定义 Inspector：
	 - 编辑条件条目（Condition 引用/选择/组合：单条件、取反、All/Any），并在内部以可编辑表单或折叠视图渲染 Condition 的字段，以规避 Tri 对内嵌展开的限制（必要时直接使用 SerializedObject + PropertyField 或自定义 IMGUI）。
	 - 在分支块内追加/插入/删除任意 `BaseNodeData` 子项（动作/跳转/条件等），并保证这些子项能被序列化回 `EventSequence.commands`。
	 - 可视化标注 If 的开始/结束位置与对应标签名（用于预编译阶段映射使用）。

3. 子类字段与布局一致性
   - 为常用 Data（ModifyItemData / PlaySfxData / LabelData / JumpData / IfData / 各 ConditionData）补齐字段标签、Tooltip、默认值，保证 Tri 展示一致性。

4. 编辑器导出与验证
   - 提供“验证”按钮/命令：在 Editor 下对当前 `EventSequence` 进行静态检查，至少校验：
	 - Jump 指向的 Label 是否存在（警告/错误并包含序列索引与标签名）。
	 - Label 重名检测（错误）。
	 - IfData 条目条件是否合法（字段完整性检查）。
	 - 检测可能导致死循环的直观跳转（静态启发式检查，不能替代运行时检测）。
   - 保存时保证 IfData 带有足够信息供 Phase4 或预编译脚本识别并展开为 `Label + BranchJump + EndJump` 的结构。

5. 可拓展性
   - TriInspectorBridge 提供小型 API/属性集，方便后续 UI Toolkit 面板直接复用 Tri 的绘制逻辑和 IfEditorBridge 的数据构建函数。

## 6. 验收效果（验收标准，更新）
- 多态创建：在 Inspector 中能为 `EventSequence.commands` 直接创建任意 `BaseNodeData` 子类实例（已验证）。
- 字段展开：创建后能展开并编辑子类字段，字段显示清晰、带 Tooltip/默认值（顶层已通过，If 内嵌条件仍需 IfEditorBridge 支持）。
- If 编辑器：在 Inspector 中创建 `IfData` 后，IfEditorBridge 能编辑条件组并在分支中插入任意 Data，保存后 Sequence 在运行时可被 EventRunner 正确加载（If 内条件编辑为当前关键阻塞点）。
- 静态校验：执行编辑器验证后，Jump 指向缺失/Label 重名会产生可定位的错误信息（包含序列索引与标签名）。
- 与 Phase2 兼容：使用编辑器编辑并保存的 Sequence 在运行时（现有 EventRunner）能被正常加载并执行（不抛出映射/反序列化异常）。

## 7. 风险与缓解（补充）
- 风险：Tri 插件在不同 Unity 版本或项目配置下差异导致展示异常。  
  缓解：封装 Tri 的调用点（TriInspectorBridge），并提供无 Tri 的回退路径（最小化功能：只显示序列摘要与只读字段）。
- 风险：IfEditorBridge 复杂性超出预期（编辑器性能、跨序列引用等）。  
  缓解：采用增量实现策略——先实现单个 If 的本地编辑再支持跨序列引用与复杂组合；限制一次编辑的最大可插入项数量以防止卡顿。
- 风险：Jump 节点表达式语法与运行时不匹配。  
  缓解：在实现前先与运行时接口（EventRunnerService）定义明确的表达式序列化协议，并通过单元测试与示例 Sequence 验证两端一致性。
